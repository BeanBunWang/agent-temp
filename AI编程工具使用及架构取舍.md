# AI 工具使用与架构取舍

## 如何使用 AI 编程工具

我用 Codex 读取需求文档和参考架构文档，使用plan模型先做出整个系统的架构，拆出必须交付的能力：Agent Loop、Tool、Skill、token 预算、边界处理、trace、README 和样例运行结果。接着按照plan生成代码与文档，再通过单元测试、真实 DeepSeek 运行和 trace 检查来验证结果。

为了效果真实，接入了DeepSeek的真实key。

执行期间遇到了执行时间过长的问题，我查看了 trace：耗时主要来自真实模型调用；文件读取、目录检索、写文件基本是 0-10ms。最终统一更换为 `deepseek-v4-flash`，并用新的 trace 重新记录耗时表现：本次运行总耗时约 102 秒，模型调用合计约 101.9 秒，较之前约 191 秒的运行明显缩短，但瓶颈仍是多轮模型请求。

## 架构判断

- 选择单 Agent 显式循环，不做多 Agent。这个题目重点是可控推进、终止条件和可复盘 trace，多 Agent 会增加状态同步和冲突处理成本。
- Tool 统一返回结构化 `ToolResult`。成功、失败、拒绝、超时都进入 trace，避免异常路径丢证据。
- Skill 只承载流程知识，启动时只加载元数据，正文按需加载，避免一开始占用过多上下文。
- 上下文压缩只替换模型看到的上下文投影，不替换真实运行状态；压缩后仍保留目标、约束、进度、重要工具结果和失败计数。
- 受限 shell 使用 allowlist、workspace cwd、超时和高风险命令拒绝。示例 trace 中 `rm -rf data` 被拒绝，用来证明边界处理有效。

## 关键取舍

- 使用 Python 标准库实现，不增加依赖，便于评审复现。
- 默认保留 `mock` provider 作为离线兜底，但交付样例使用真实 DeepSeek。
- 真实模型仍可能因为多轮决策变慢；如果要进一步提速，优先减少模型轮次或关闭边界验证步骤，而不是继续优化文件读取。
- 不实现长期记忆、提醒创建、MCP 和多 Agent；这些能力需要更复杂的权限、审批、恢复和版本治理，不适合塞进第一版。
